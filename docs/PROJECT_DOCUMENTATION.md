# Acmebot for Azure — Project Documentation

## Overview

Acmebot is an **automated SSL/TLS certificate management system** for Azure. It uses the ACME protocol (the same one behind Let's Encrypt) to automatically issue and renew certificates, storing them in **Azure Key Vault**. It runs as an **Azure Functions** app using **Durable Functions** for reliable orchestration.

**Key Goals:**

- Centrally manage certificates via Azure Key Vault
- Automate certificate issuance and renewal
- Support multiple DNS providers for DNS-01 ACME challenge validation
- Provide high reliability with monitoring and webhooks
- Support multiple Azure services (App Service, Container Apps, Application Gateway, Front Door, CDN, etc.)
- Support multiple ACME CAs (Let's Encrypt, ZeroSSL, Google Trust Services, SSL.com, Entrust)

## ACME Protocol Client

### How `AcmeClient` Works

1. **Account Management**
   - `CreateAccountAsync()`: Registers new ACME account with optional External Account Binding (EAB)
   - `FindAccountAsync()`: Retrieves existing account
   - `UpdateAccountAsync()`: Updates account contact info

2. **Order Lifecycle**
   - Fetches ACME directory from server
   - Creates orders for domain names
   - Manages challenge responses
   - Finalizes orders with certificate signing requests (CSR)
   - Downloads certificate chains

3. **Signing & Cryptography (`AcmeSigner`)**
   - Supports **ECDSA** (ES256, ES384, ES512) and **RSA** (RS256, RS384, RS512)
   - Uses JWS (JSON Web Signature) format for ACME protocol messages
   - Generates thumbprints for ACME account keys
   - Uses RFC 8037 format for ECDSA curves

4. **Error Handling**
   - `AcmeProtocolException` encapsulates ACME server responses
   - Includes problem details, HTTP status codes, retry-after headers
   - Supports bad nonce retry logic for protocol compliance

## Certificate Issuance Workflow

The heart of the system is a **Durable Functions orchestrator** (`CertificateIssuanceOrchestrator`) that follows this flow:

```
 1. Dns01Precondition      → Validate DNS zone exists and is reachable
 2. Order                  → Create ACME order for domain(s)
 3. Dns01Authorization     → Get DNS-01 challenge instructions
 4. CreateTimer            → Wait for DNS propagation (configurable per provider)
 5. CheckDnsChallenge      → Verify TXT records are resolvable
 6. AnswerChallenges       → Tell ACME CA to validate
 7. CheckIsReady           → Poll until validation completes
 8. CleanupDnsChallenge    → Delete temporary TXT records
 9. FinalizeOrder          → Submit CSR (generated by Key Vault)
10. CheckIsValid           → Poll until certificate is issued
11. MergeCertificate       → Merge cert chain into Key Vault
12. SendCompletedEvent     → Webhook notification
```

The orchestrator uses a retry policy of **12 retries with 5-second intervals** for `RetriableActivityException`.

## Automatic Certificate Renewal

### Daily Timer Trigger

In `RenewCertificates.cs`, the `Timer` function fires daily at midnight UTC:

```csharp
[Function($"{nameof(RenewCertificates)}_{nameof(Timer)}")]
public async Task Timer([TimerTrigger("0 0 0 * * *")] TimerInfo timer, [DurableClient] DurableTaskClient starter)
{
    var instanceId = await starter.ScheduleNewOrchestrationInstanceAsync(
        $"{nameof(RenewCertificates)}_{nameof(Orchestrator)}");
}
```

The CRON expression `"0 0 0 * * *"` means **every day at 00:00:00 UTC**. This is hardcoded and not user-configurable.

### Determining Which Certificates Need Renewal

In `CertificateActivities.cs`, the `GetRenewalCertificates` activity iterates all Key Vault certificates and selects those due for renewal using two strategies:

**Strategy A — ACME Renewal Info API:** If the CA supports the Renewal Info endpoint, it checks whether the current time has passed the `SuggestedWindow.Start` from the CA.

**Strategy B — Expiry-based fallback:**

```csharp
if ((properties.ExpiresOn.GetValueOrDefault(DateTimeOffset.MaxValue) - now).TotalDays <= _options.RenewBeforeExpiry)
{
    result.Add(certificate.Value.ToCertificateItem());
}
```

This compares the days until expiry against the configurable `RenewBeforeExpiry` option (default: 30 days).

Only certificates tagged as issued by Acmebot and matching the current ACME endpoint are considered.

### Random Jitter (Thundering Herd Prevention)

After finding certificates to renew, a random delay of **0–600 seconds (0–10 minutes)** is added:

```csharp
var jitter = (uint)context.NewGuid().GetHashCode() % 600;
await context.CreateTimer(context.CurrentUtcDateTime.AddSeconds(jitter), CancellationToken.None);
```

This uses `context.NewGuid()` (deterministic in Durable Functions replay) to generate a stable random value, ensuring all instances don't hit the ACME CA simultaneously. The 600-second window is hardcoded.

### Renewal Retry Policy

The renewal orchestrator uses a retry policy of **2 retries with 3-hour backoff**, only for `RetriableOrchestratorException`. If a certificate renewal fails, the error is logged and the orchestrator continues with the next certificate.

### Configuration

The renewal window is defined in `AcmebotOptions.cs`:

```csharp
[Range(0, 365)]
public int RenewBeforeExpiry { get; set; } = 30;
```

Users set this via application settings on the Function App (or `local.settings.json`):

```json
{
  "Acmebot:RenewBeforeExpiry": 45
}
```

### Configurable vs Hardcoded Renewal Behavior

| Behavior | Configurable? | Value |
|---|---|---|
| Renewal window (days before expiry) | **Yes** — `Acmebot:RenewBeforeExpiry` | Default 30 |
| Timer schedule (daily midnight) | **No** — hardcoded CRON | `0 0 0 * * *` (every day at midnight UTC) |
| Jitter range | **No** — hardcoded | 0–600 seconds |
| Retry policy on failure | **No** — hardcoded | 2 retries, 3-hour backoff |

## Azure Key Vault Integration

- Key Vault generates the **private key and CSR** — the private key never leaves Key Vault.
- After ACME issues the certificate, it's **merged** back into Key Vault with the private key.
- Certificates are tagged with ACME metadata (endpoint URL, "issued by Acmebot" marker, DNS names) for tracking.

|---|---|---|---|

Each provider is configured under its own section (e.g., `Acmebot:AzureDns`, `Acmebot:Cloudflare`, etc.) with provider-specific options like API keys, subscription IDs, or credentials.

## Design Patterns

1. **Durable Functions** — Reliable, long-running certificate workflows with automatic retries and replay safety.
2. **Pluggable DNS Providers** — Common `IDnsProvider` interface with DI registration; easy to add new providers.
3. **Exception Hierarchy** — `PreconditionException` (non-retriable config errors) vs. `RetriableActivityException` / `RetriableOrchestratorException` (transient failures).
4. **Preferred Chain Selection** — Allows choosing certificate chain (e.g., ISRG Root X1 vs. DST Root CA X3).
5. **External Account Binding** — Support for CAs requiring EAB (ZeroSSL, Google Trust Services, SSL.com).
6. **Account State Management** — `AcmeClientFactory` stores account details in file system, caches keys, and validates/updates contacts automatically.
7. **Application Insights Integration** — Structured logging, request tracking, and webhook delivery monitoring.
